## 개발 프로세스

### 1. 작업 흐름 (Workflow)

```
Task 생성 (Product Owner)
  ↓
Task 할당 (분대장)
  ↓
개발 시작 (병사)
  ↓
코드 리뷰 (분대장)
  ↓
Staging 배포
  ↓
통합 테스트
  ↓
Production 배포 (소대장 승인)
  ↓
모니터링
```

### 2. Git 워크플로우 (GitHub Flow)

#### 브랜치 명명 규칙
```
feature/설명             # 새로운 기능
bugfix/설명              # 버그 수정
hotfix/설명              # 긴급 수정
refactor/설명            # 리팩토링
docs/설명                # 문서 작성
```

#### 예시
```
feature/ai-chat-widget
bugfix/dashboard-chart-loading
hotfix/login-sql-injection
```

#### Pull Request 프로세스

1. **브랜치 생성 및 개발**
```bash
git checkout -b feature/새로운-기능
# 개발 진행
git add .
git commit -m "feat: 설명"
git push origin feature/새로운-기능
```

2. **PR 생성**
```
Title: 기능 설명 (30자 이내)
Description:
- 변경 사항
- 관련 이슈
- 테스트 결과
```

3. **코드 리뷰**
- 분대장이 검토
- 최소 1명 승인 필요

4. **병합 및 배포**
```bash
git merge --squash feature/새로운-기능
git push origin main
# 자동 배포 트리거
```

### 3. 배포 프로세스

#### Staging 배포 (자동)
```
PR 승인 → Staging 배포 → 테스트
```

#### Production 배포 (수동)
```
소대장 승인 → Main 브랜치 병합 → 자동 배포 → 모니터링
```

#### 배포 전 체크리스트
- [ ] 모든 테스트 통과
- [ ] 코드 리뷰 완료
- [ ] 문서 업데이트
- [ ] 마이그레이션 준비
- [ ] 롤백 계획 수립

## 코드 리뷰 가이드라인

### 1. 리뷰어 책임

#### 검토 항목
- [ ] 코드 정확성 (버그 없음)
- [ ] 성능 (불필요한 루프, 쿼리 등)
- [ ] 보안 (SQL injection, XSS 등)
- [ ] 코드 스타일 일관성
- [ ] 문서화 및 주석
- [ ] 테스트 커버리지 (최소 70%)

#### 리뷰 의견 작성
```
좋은 리뷰:
"이 부분의 SQL 쿼리가 N+1 문제를 가지고 있습니다.
다음과 같이 JOIN을 사용하면 성능이 개선됩니다:
[예시 코드]"

안 좋은 리뷰:
"이건 안 돼요" (이유 없음)
```

### 2. 리뷰 대상자 책임

#### 피드백 대응
1. 질문 이해하기
2. 해당 부분 개선하기
3. 변경 사항 설명하기
4. 재리뷰 요청하기

#### 리뷰 거절
```
리뷰가 부당하다고 생각되면:
1. 존중하는 톤으로 설명
2. 근거 제시
3. 논의 후 합의
```
